    **Theory files (6):**
    Each theory file must include "Why This Matters" with mobile-dev analogies and "Key Takeaways."

    1. `01-why-version-apis.md` -- Why APIs need versioning, real-world examples of breaking changes, the cost of not versioning (every mobile app update breaks), API contract concept, semantic versioning for APIs. Mobile analogy: iOS deployment target (15, 16, 17), Android minSdkVersion -- same idea of supporting old clients.

    2. `02-url-path-versioning.md` -- URL path versioning with FastAPI APIRouter prefix (/v1/users, /v2/users), separate routers per version, shared business logic between versions, OpenAPI docs separation by tags. Show complete example with v1 returning flat response, v2 returning wrapped response with metadata. Use native FastAPI -- no third-party library (fastapi-versioning is poorly maintained).

    3. `03-header-versioning.md` -- Header-based versioning using X-API-Version custom header, FastAPI Header() dependency for extraction, version dispatching within a single endpoint, default version when header not provided, 400 for unsupported versions. Show example with Header(default="1.0") pattern. When to use header vs URL versioning (URL is more explicit and preferred; header keeps URLs clean but is less discoverable).

    4. `04-breaking-vs-nonbreaking.md` -- Clear definitions: BREAKING = removing a field, changing a field type, removing an endpoint, changing a required parameter. NON-BREAKING = adding an optional field, adding a new endpoint, adding optional query parameter. Decision framework for when to version. Common mistake: versioning for every change.

    5. `05-deprecation-notices.md` -- HTTP Sunset header (RFC 8594), Deprecation header, Link header with rel="successor-version", deprecation timeline best practices, communicating to API consumers, logging deprecated endpoint usage for tracking. Show complete FastAPI response header implementation.

    6. `06-maintaining-versions.md` -- Strategies for maintaining multiple versions: shared service layer with version-specific routers, adapter pattern for transforming responses, testing both versions, sunset timeline and removal process. When to drop a version. Mobile analogy: maintaining backward compatibility across multiple OS versions.

    **Exercises (3):**
    All exercises use FastAPI TestClient. Exercises are self-contained Python files.

    1. `01_url_versioning.py` -- Exercise: Create URL path versioning. Learner must: create v1_router and v2_router with APIRouter(prefix="/v1") and "/v2", implement GET /users/{user_id} on both routers with different response shapes (v1: flat {"id", "name"}, v2: wrapped {"data": {"id", "name", "email"}, "meta": {"version"}}), include both routers in the app. TODO stubs for router creation and endpoint functions. Tests verify: GET /v1/users/1 returns flat response, GET /v2/users/1 returns wrapped response with meta, both return 200.

    2. `02_header_versioning.py` -- Exercise: Implement header-based versioning. Learner must: create a single GET /users/{user_id} endpoint that reads X-API-Version header (default "1.0"), return different response shapes based on version, return 400 for unsupported versions. TODO stubs for the endpoint function. Tests verify: no header returns v1 format, X-API-Version: 1.0 returns v1, X-API-Version: 2.0 returns v2 format, X-API-Version: 3.0 returns 400.

    3. `03_deprecation.py` -- Exercise: Add deprecation headers. Learner must: create v1 and v2 routers, v1 GET /users/{user_id} adds Deprecation: true, Sunset date header, and Link header pointing to v2 successor, v2 endpoint returns normal response without deprecation headers. TODO stubs for both endpoints. Tests verify: v1 response has Deprecation header set to "true", has Sunset header with valid date, has Link header with rel="successor-version", v2 response does NOT have Deprecation header.

    **Project README:**

Include starter template and success criteria.
