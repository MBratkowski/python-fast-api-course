    **Theory files (6):**
    Each theory file must include "Why This Matters" with mobile-dev analogies and "Key Takeaways."

    1. `01-rate-limiting-algorithms.md` -- Why rate limit (prevent abuse, ensure fairness, protect resources), algorithm overview: fixed window (simple but boundary burst issue), sliding window (precise but more memory), token bucket (burst-friendly with refill), leaky bucket (smooth output). Visual diagrams using ASCII art. Mobile analogy: iOS URLSession rate limiting, Android OkHttp rate limiting interceptors.

    2. `02-redis-implementation.md` -- Why Redis for rate limiting (atomic operations, fast, shared across instances), token bucket implementation with Redis GET/SET and pipeline for atomicity, sliding window with sorted sets (ZADD to add timestamp, ZREMRANGEBYSCORE to remove old entries, ZCARD to count), key expiry for auto-cleanup. Show complete implementations of both algorithms. Warn about race conditions with non-atomic read-check-increment.

    3. `03-per-user-vs-per-ip.md` -- Different rate limit keys: IP address (request.client.host), authenticated user ID (from JWT), API key, combined keys. Per-user limits allow different tiers (free: 100/hour, pro: 1000/hour). IP-based limits for unauthenticated endpoints. Extracting user identity from JWT token. Mobile analogy: app store API rate limits per developer account.

    4. `04-monthly-quotas.md` -- Long-term usage tracking with Redis counters, monthly reset logic, quota tiers (free/pro/enterprise), tracking usage across multiple endpoints, quota percentage warnings (80%, 90%, 100%). INCR with EXPIREAT for monthly counters. Difference between rate limiting (short-term burst protection) and quotas (long-term usage caps).

    5. `05-response-headers.md` -- Standard rate limit headers: X-RateLimit-Limit (max requests), X-RateLimit-Remaining (requests left), X-RateLimit-Reset (unix timestamp when window resets), Retry-After (seconds to wait on 429). Adding headers via FastAPI middleware or Response object. 429 Too Many Requests status code with JSON error body.

    6. `06-client-side-handling.md` -- How mobile clients should handle rate limits: read Retry-After header, implement exponential backoff, queue requests during rate limit, show user-friendly messages. 429 response handling pattern. Mobile analogy: how iOS/Android apps handle 429 from backend APIs they consume (URLSession retry, Retrofit retry interceptor).

    **Exercises (3):**
    All exercises use fakeredis (redis.asyncio compatible) and are self-contained with embedded pytest tests. Import fakeredis.aioredis for async Redis mock. Use pytest-asyncio for async tests.

    1. `01_token_bucket.py` -- Exercise: Implement token bucket rate limiter. Provide TokenBucketLimiter class skeleton with __init__(redis_client, capacity, refill_rate) and async is_allowed(key) -> bool method as TODO stubs. Tests verify: first N requests (up to capacity) are allowed, request N+1 is denied, after waiting (simulated via time manipulation), tokens refill and requests are allowed again. Use fakeredis.aioredis.FakeRedis() as the Redis client. Tests should mock time.time() to control refill timing without actual sleeps.

    2. `02_sliding_window.py` -- Exercise: Implement sliding window rate limiter. Provide SlidingWindowLimiter class skeleton with __init__(redis_client, max_requests, window_seconds) and async is_allowed(key) -> tuple[bool, int] (allowed, remaining) as TODO stubs. Learner must use ZADD, ZREMRANGEBYSCORE, ZCARD in a Redis pipeline. Tests verify: requests within limit return (True, remaining_count), requests exceeding limit return (False, 0), old requests outside window are cleaned up. Use fakeredis and mock time.time().

    3. `03_per_user_limits.py` -- Exercise: Combine rate limiting with user identification. Provide a FastAPI app skeleton with a rate-limited endpoint that extracts user ID from a simple auth header and applies per-user limits using either token bucket or sliding window. TODO stubs for the rate limit check function and the endpoint. Tests verify: user A making requests does not affect user B's limit, each user has independent counters, unauthenticated requests use IP-based key. Use TestClient + fakeredis.

    **Project README:**
    "Implement Rate Limiting for an API" -- Learner adds comprehensive rate limiting to a FastAPI application. Requirements: token bucket for burst-friendly endpoints, sliding window for strict endpoints, per-user limits with JWT integration, rate limit response headers on every response, 429 responses with Retry-After, monthly quota tracking. Include starter template and success criteria.

    **Module README:**
    Overview with learning objectives, prerequisites (Redis concepts from Module 014, JWT from Module 009), and links to theory/exercises/project. Note: requires fakeredis package (already installed from Module 014).

    1. `01-when-to-use-microservices.md` -- Monolith vs microservices tradeoffs, when to decompose (team size, deployment frequency, scaling needs), premature microservices anti-pattern (start monolith, split when needed), real-world examples of monolith-first companies. Mobile analogy: single app vs app suite (Google Maps + Waze, Facebook + Messenger) -- same "service boundary" concept.

    2. `02-service-boundaries.md` -- Domain-driven design basics for finding boundaries, bounded contexts, one database per service rule, shared-nothing architecture, identifying service boundaries in a sample monolith (users, orders, products). Anti-pattern: shared database between services (tight coupling, schema changes break everything). Mobile analogy: Swift Package modules, multi-module Gradle project.

    3. `03-sync-communication.md` -- HTTP-based service-to-service calls using httpx AsyncClient, request/response pattern, timeout handling (ALWAYS set timeout), error handling (HTTPStatusError, RequestError), circuit breaker concept (theory-level), retry with exponential backoff. Show complete example: Order Service calling User Service to get user details. Anti-pattern: synchronous calls in async FastAPI without httpx (use httpx.AsyncClient, NOT requests library).

    4. `04-async-communication.md` -- Event-driven architecture, publish/subscribe pattern, Redis pub/sub for lightweight messaging, message format conventions (JSON with event_type, timestamp, payload), eventual consistency concept, when to use async vs sync communication (sync for queries, async for events/notifications). Show complete Redis pub/sub example with publisher and subscriber. Mobile analogy: NotificationCenter on iOS, EventBus/SharedFlow on Android.

    5. `05-api-gateway.md` -- What an API gateway does (routing, auth, rate limiting, aggregation), simple gateway pattern with FastAPI proxy routes, routing based on URL prefix to backend services, request forwarding with httpx, aggregation pattern (gateway calls multiple services and combines responses). Show complete gateway implementation using httpx to forward requests.

    6. `06-data-consistency.md` -- CAP theorem basics (consistency, availability, partition tolerance -- pick two), eventual consistency in microservices, saga pattern concept (sequence of local transactions with compensating actions), idempotency for safe retries, distributed transactions (avoid if possible -- use sagas instead). Keep conceptual -- this is advanced theory, not exercise material.

TODO stubs for both functions. Tests verify: published event is received by subscriber, event data is correctly serialized/deserialized as JSON, multiple events on same channel are all received.
