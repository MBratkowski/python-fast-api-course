    **Theory files (6):**
    Each theory file must include "Why This Matters" with mobile-dev analogies (Instruments/Android Profiler = cProfile, Core Data logging = SQLAlchemy echo, XCTest performance tests = Locust) and "Key Takeaways."

    1. `01-profiling-basics.md` -- Why profile before optimizing (measure, don't guess). cProfile for function-level profiling. line_profiler for line-by-line analysis. Reading profiler output (cumulative time, call count). pstats for sorting and filtering results. Show complete profiling workflow: profile -> identify bottleneck -> optimize -> re-profile.
    2. `02-query-analysis.md` -- SQLAlchemy echo mode for development. EXPLAIN and EXPLAIN ANALYZE for PostgreSQL. Reading query plans (Seq Scan vs Index Scan, nested loops). Index creation strategies. When to add indexes (columns in WHERE, JOIN, ORDER BY). Common mistake: profiling Python code when the bottleneck is database I/O.
    3. `03-n-plus-one-queries.md` -- What the N+1 problem is (1 query for list + N queries for relationships). How SQLAlchemy lazy loading causes it. Fixes: `selectinload()` for one-to-many (SELECT ... WHERE id IN), `joinedload()` for many-to-one (single JOIN query). When to use each. Show query count before and after. Use SQLAlchemy 2.0 select() style exclusively (not legacy Query API).
    4. `04-connection-pooling.md` -- What connection pooling is (reuse database connections). SQLAlchemy's built-in pool (QueuePool). Configuration: pool_size, max_overflow, pool_timeout, pool_recycle, pool_pre_ping. Symptoms of pool exhaustion (500 errors under load). Sizing guidelines. Never use echo=True in production.
    5. `05-async-best-practices.md` -- When async helps (I/O-bound operations) vs when it doesn't (CPU-bound). Avoiding blocking calls in async endpoints. Using asyncio.gather for concurrent I/O. Thread pool executor for CPU-bound work. Common mistake: using sync database driver in async endpoint. Connection between Module 012 async concepts and production optimization.
    6. `06-load-testing.md` -- Why load test (find breaking points before users do). Locust introduction: Python-native, HttpUser, @task decorator, wait_time. Running Locust (web UI and headless mode). Reading results (RPS, response times, percentiles). Configuring user count, spawn rate, run time. Generating HTML reports. Writing realistic test scenarios.

    1. `01_profile_code.py` -- Provide a deliberately inefficient Python function (e.g., nested loops, redundant computations, string concatenation in loop). Learner must use cProfile to identify the bottleneck and then implement an optimized version. TODO stubs for: running the profiler, writing the optimized function. Tests verify the optimized function produces the same output and is measurably faster (using time comparison or call count reduction).
    2. `02_fix_n_plus_one.py` -- Provide SQLAlchemy 2.0 models (e.g., Author with Posts, using DeclarativeBase, Mapped, mapped_column) with an in-memory SQLite database pre-seeded with test data. Provide a function that has N+1 queries (iterates posts and accesses post.author). Learner must rewrite the query using selectinload/joinedload. TODO stubs for the optimized query functions. Tests verify: correct results returned AND query count is reduced (use SQLAlchemy event listeners to count queries, or compare with expected result).
    3. `03_load_test.py` -- Exercise: Write a Locust test file. Provide a simple FastAPI app (in the exercise). Learner must write a Locust HttpUser class with @task decorators for different endpoints with appropriate weights and wait_time. TODO stubs for the Locust user class. Tests validate the Locust file structure: has HttpUser subclass, has wait_time, has @task decorated methods, has appropriate endpoint URLs. (Tests parse and validate the class structure, not actually run load tests.)
