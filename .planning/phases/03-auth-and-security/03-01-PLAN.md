    **1. `01-authn-vs-authz.md` — Authentication vs Authorization**
    Cover: Authentication = proving who you are (login), Authorization = proving what you can do (permissions). The flow: client sends credentials -> server verifies -> issues token -> client sends token on every request -> server validates token. HTTP is stateless — tokens make it stateful per request. Where auth fits in the request lifecycle. Analogy: "Authentication is like unlocking your phone with Face ID — proving you are you. Authorization is like app permissions — you're logged in, but can you access the camera? In mobile, you handle the client side (storing tokens in Keychain/Keystore, sending Bearer headers). Now you build the server side." Show the complete flow diagram as text.

    **2. `02-password-hashing.md` — Password Hashing**
    Cover: Why never store plain-text passwords. What hashing is (one-way function). Why MD5/SHA are NOT for passwords (too fast, rainbow tables). Modern password hashing: Argon2 (memory-hard, recommended), bcrypt (time-tested), scrypt. Using `pwdlib` with Argon2: `PasswordHash.recommended()`, `hash()`, `verify()`. Salt (automatic with pwdlib). Timing attack prevention: always hash even for invalid usernames (use DUMMY_HASH pattern from research). Analogy: "In mobile, you send the password to the server and trust it handles hashing. Now YOU are that server. One breach of plain-text passwords and your users are compromised everywhere they reused that password." Show: hash a password, verify a password, the timing attack prevention pattern.

    **3. `03-jwt-structure.md` — JWT Structure and Claims**
    Cover: What is JWT (JSON Web Token) — three parts: header.payload.signature. Header: algorithm and token type. Payload: claims (sub, exp, iat, custom claims). Signature: HMAC-SHA256 of header+payload with secret key. JWT is signed NOT encrypted — anyone can read the payload (base64), but only the server can create valid signatures. Standard claims: sub (subject/user_id), exp (expiration), iat (issued at), jti (unique token ID). Using PyJWT: `jwt.encode(payload, SECRET_KEY, algorithm="HS256")`, `jwt.decode(token, SECRET_KEY, algorithms=["HS256"])`. ALWAYS specify algorithms list (prevent algorithm confusion attacks). Generate secrets with `openssl rand -hex 32`. Analogy: "You've decoded JWTs in your mobile app (maybe to check expiration before making a request). The payload is just base64-encoded JSON. Now you understand why — JWT is transparent by design, security comes from the signature." Show: create a token, decode it, handle expired/invalid tokens.

    **4. `04-access-refresh-tokens.md` — Access and Refresh Tokens**
    Cover: Why two tokens? Access token: short-lived (15-30 min), sent on every request, contains user identity. Refresh token: long-lived (7-14 days), used ONLY to get new access tokens, stored securely. The refresh flow: access token expires -> client sends refresh token -> server validates and issues new access token -> (optionally) rotate refresh token. Why short-lived access tokens: limits damage from stolen tokens. Token rotation: each refresh invalidates the old refresh token (detects theft). Analogy: "In mobile, you implement this flow: store refresh token in Keychain/Keystore, store access token in memory, refresh silently when 401. Now you build the server that issues and validates both tokens." Show: create_access_token and create_refresh_token functions, the refresh endpoint flow.

    **5. `05-token-expiration-rotation.md` — Token Expiration and Rotation**
    Cover: Setting expiration with `timedelta` and the `exp` claim. PyJWT automatically validates `exp` on decode (raises `ExpiredSignatureError`). Handling expired tokens gracefully (return 401 with clear message). Refresh token rotation: issue new refresh token on each refresh, invalidate the old one. Token revocation strategies: Redis blacklist with TTL, database storage with is_revoked flag. Logout: revoke refresh token. Force-logout all sessions: change user's token version/secret. Analogy: "This is like managing session tokens in your mobile app's auth manager — but server-side. You decide the lifecycle: when tokens are born, when they expire, and when they die." Show: token expiration handling, refresh with rotation pseudocode, logout endpoint.

    **6. `06-oauth2-password-bearer.md` — FastAPI OAuth2PasswordBearer**
    Cover: FastAPI's security module — built-in OAuth2 support. `OAuth2PasswordBearer(tokenUrl="auth/login")` — declares that tokens come from a login endpoint. How it works: extracts `Authorization: Bearer <token>` header automatically. `OAuth2PasswordRequestForm` — standard form for username+password (NOT JSON — OAuth2 spec requires form data). Building `get_current_user` dependency: extract token -> decode JWT -> load user from database -> return user or raise 401. Composing dependencies: `get_current_user` depends on `oauth2_scheme` depends on request header. OpenAPI integration: Swagger UI gets a login button automatically. Analogy: "OAuth2PasswordBearer is like your mobile app's auth interceptor (OkHttp Interceptor, Alamofire RequestAdapter) — it automatically extracts and validates the token from every request. But here it's server-side, using FastAPI's dependency injection." Show: complete OAuth2PasswordBearer setup, get_current_user dependency, login endpoint, protected endpoint.

    **1. `01-rbac-concepts.md` — RBAC Concepts**
    Cover: What is RBAC (Role-Based Access Control) — assign roles to users, assign permissions to roles, check user's role at each endpoint. Common role models: flat roles (admin, user, moderator), hierarchical roles (admin > moderator > user), permission-based (read, write, delete). RBAC vs ABAC (attribute-based) — RBAC is simpler and sufficient for most APIs. Where authorization fits: after authentication (first prove who you are, then check what you can do). The principle of least privilege: users get minimum permissions needed. Analogy: "In mobile, you check permissions before accessing the camera or location. RBAC is the same concept for your API — can this user access this endpoint? In iOS, you check `AVCaptureDevice.authorizationStatus`; in your API, you check `user.role`." Show: flat role model diagram, the flow from request -> authenticate -> authorize -> handle.

    **2. `02-roles-permissions.md` — Defining Roles and Permissions**
    Cover: Modeling roles in Python: `Role(str, Enum)` with values ADMIN, USER, MODERATOR. Adding role to User model: `role: Mapped[Role] = mapped_column(SQLEnum(Role), default=Role.USER)`. Permissions as a separate concept: roles have permissions (admin can delete any user, moderator can ban users, user can edit own profile). Simple approach: check role directly. Advanced approach: role -> permission mapping dict. Default roles on signup (always USER unless explicitly set). Analogy: "This is like defining user types in your mobile app's backend — free users vs premium vs admin. In your mobile app, you show/hide UI based on the user type from the API. Now you enforce it server-side." Show: Role enum, User model with role field, role-permission mapping.

    **3. `03-permission-dependencies.md` — Permission Dependencies in FastAPI**
    Cover: The `require_role()` dependency factory pattern — returns a dependency function that checks the user's role. Using it: `Depends(require_role(Role.ADMIN))`. The `require_any_role()` pattern for endpoints accessible by multiple roles. Composing dependencies: get_current_user -> require_role -> endpoint. How FastAPI's dependency chain works for auth: oauth2_scheme -> get_current_user -> require_role. Error handling: return 403 Forbidden (not 401 — user is authenticated but not authorized). Analogy: "Dependencies in FastAPI are like middleware interceptors in your mobile networking stack — each one checks a condition and either passes through or blocks. `require_role` is like a route guard in Angular or a NavigationGuard in Vue." Show: require_role factory, require_any_role factory, usage on endpoints, the full dependency chain.

    **4. `04-resource-authorization.md` — Resource-Level Authorization**
    Cover: Beyond role checks — does this user OWN this resource? The confused deputy problem: authenticated user accessing another user's data. Resource ownership dependency: `get_post_or_403(post_id, current_user)` — fetches resource, checks if current_user is the owner (or admin). Pattern: fetch resource -> check owner_id == current_user.id OR current_user.role == ADMIN -> return resource or raise 403. Use dependencies (not middleware) for resource-level auth — you need the resource ID from the path. Analogy: "In mobile, you only show edit/delete buttons for the user's own content. But a malicious user could call the API directly. Resource-level auth prevents 'just change the user_id in the request' attacks." Show: get_post_or_403 dependency, usage on update/delete endpoints, admin bypass.

    Cover: Admin-specific endpoints: user management (list all users, activate/deactivate, change roles), system stats, bulk operations. Protecting entire routers: use `dependencies=[Depends(require_role(Role.ADMIN))]` on `APIRouter()`. Admin user creation: seed admin via script or environment variable (never through public signup). Preventing privilege escalation: non-admin cannot set role to ADMIN. Audit logging for admin actions.
