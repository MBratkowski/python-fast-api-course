    **1. `01-relational-concepts.md` — Relational Database Concepts**
    Cover: What is a relational database, tables/rows/columns, primary keys (auto-increment), data types (INTEGER, VARCHAR, TEXT, BOOLEAN, TIMESTAMP), NULL vs NOT NULL, UNIQUE constraints, DEFAULT values. Analogy: "You've used Core Data entities or Room @Entity — a SQL table is the same idea, but the database enforces the schema instead of your app code." Compare: table = entity, row = object instance, column = property, primary key = unique ID. Show CREATE TABLE with common types.

    **2. `02-sql-crud.md` — SQL CRUD Operations**
    Cover: INSERT INTO (single and multiple rows), SELECT with WHERE, UPDATE with SET and WHERE, DELETE with WHERE. Cover SELECT variations: WHERE with comparison operators (=, !=, <, >, LIKE, IN, BETWEEN, IS NULL), ORDER BY (ASC/DESC), LIMIT and OFFSET, COUNT/SUM/AVG/MIN/MAX aggregate functions, GROUP BY with HAVING. Analogy: "In your mobile app, you call `dao.getAll()` or `fetchRequest` — here you write the actual query those ORMs generate." Show each operation with a users table example. Emphasize: always use WHERE with UPDATE/DELETE (or you affect all rows).

    **3. `03-relationships.md` — Table Relationships**
    Cover: One-to-one (user ↔ profile), one-to-many (user → posts), many-to-many (students ↔ courses with junction table). Foreign keys: REFERENCES syntax, ON DELETE CASCADE/SET NULL/RESTRICT. Normalization principles: avoid data duplication, separate concerns into tables, use foreign keys to connect. Analogy: "In mobile, you nest objects (user.profile) or use IDs to reference related objects. In SQL, foreign keys formalize these references and the database enforces them." Show CREATE TABLE with FOREIGN KEY for each relationship type. Include the junction/pivot table pattern for many-to-many.

    **4. `04-joins.md` — Joins Explained**
    Cover: INNER JOIN (matching rows from both), LEFT JOIN (all left + matching right), RIGHT JOIN (all right + matching left), FULL OUTER JOIN, CROSS JOIN (cartesian product — mention but discourage). Use a visual text diagram for each join type showing which rows are included. Practical examples: "Get all users with their posts" (LEFT JOIN — includes users without posts), "Get only users who have posted" (INNER JOIN). Show multi-table joins. Analogy: "JOINs are like combining two arrays based on a matching ID — like merging a user list with their order list by user_id." Cover aliasing (u for users, p for posts).

    **5. `05-indexes-performance.md` — Indexes and Performance**
    Cover: What is an index (like a book's index — lookup without scanning every page), CREATE INDEX syntax, when to add indexes (columns in WHERE, JOIN, ORDER BY), when NOT to index (small tables, rarely queried columns, columns with low cardinality), B-tree index (default), composite indexes, UNIQUE indexes. Cover EXPLAIN to analyze query plans. Analogy: "Think of it like adding a search index to your mobile app's local database — Core Data has indexes on properties, Room has @Index." Show before/after EXPLAIN output. Warn about over-indexing (slower writes, more storage).

    **6. `06-postgresql-setup.md` — PostgreSQL Setup and psql**
    Cover: Why PostgreSQL for backend development (most popular for Python APIs, ACID compliance, rich feature set). Installing via Docker Compose (primary method): show docker-compose.yml with PostgreSQL service, environment variables (POSTGRES_USER, POSTGRES_PASSWORD, POSTGRES_DB), port mapping, volume for data persistence. Brief mention of local installation (Homebrew/apt) as alternative. Using psql CLI: connecting (`psql -h localhost -U user -d dbname`), common commands (\dt list tables, \d table describe table, \q quit, \l list databases), executing SQL files (\i file.sql). Analogy: "Docker Compose is like your Android emulator or iOS Simulator — it runs the database in an isolated environment so you don't pollute your machine." Show the complete docker-compose.yml and common psql workflow.

Exercises are pure Python (no SQLAlchemy, no database connection) — they test SQL knowledge using string-based SQL and Python data structures, matching the Phase 1 pattern where Module 002 exercises were pure Python to focus on concepts.

    **Exercise files (3 files):**
    Each has a docstring header with `Run: pytest 006-sql-databases/exercises/...` command, TODO stubs with `pass` or minimal returns, and inline pytest tests after `# ============= TESTS =============`.

    **1. `01_sql_queries.py` — Writing and Analyzing SQL Queries**
    Functions that generate or analyze SQL queries as strings and work with data:

    - `build_select_query(table: str, columns: list[str], where: dict | None = None) -> str` — Build a SELECT query string. Example: `build_select_query("users", ["name", "email"], {"is_active": True})` → `"SELECT name, email FROM users WHERE is_active = true"`. Handle None where (no WHERE clause). Handle multiple where conditions joined with AND.

    - `build_insert_query(table: str, data: dict) -> str` — Build an INSERT query string. Example: `build_insert_query("users", {"name": "Alice", "email": "a@b.com"})` → `"INSERT INTO users (name, email) VALUES ('Alice', 'a@b.com')"`. String values get single quotes, numbers don't.

    - `parse_where_clause(clause: str) -> list[dict]` — Parse a WHERE clause string into conditions. Example: `parse_where_clause("age > 18 AND status = 'active'")` → `[{"column": "age", "operator": ">", "value": "18"}, {"column": "status", "operator": "=", "value": "active"}]`. Handle operators: =, !=, >, <, >=, <=.

    - `calculate_query_result(rows: list[dict], query_type: str, column: str | None = None) -> int | float` — Simulate aggregate functions on in-memory data. query_type is one of "COUNT", "SUM", "AVG", "MIN", "MAX". For "COUNT", column can be None (count all rows). For others, column specifies which field to aggregate.

    Include 6-8 pytest tests covering normal cases and edge cases (empty where, multiple conditions, numeric vs string values, empty rows for aggregates).

    **2. `02_schema_design.py` — Designing Database Schemas**
    Functions that return schema designs as Python data structures:

    - `design_user_schema() -> dict` — Return a dict representing a users table schema: `{"table": "users", "columns": [...], "constraints": [...]}`. Each column: `{"name": str, "type": str, "nullable": bool, "default": value | None}`. Must include: id (INTEGER, PRIMARY KEY), username (VARCHAR(50), UNIQUE, NOT NULL), email (VARCHAR(255), UNIQUE, NOT NULL), created_at (TIMESTAMP, DEFAULT CURRENT_TIMESTAMP), is_active (BOOLEAN, DEFAULT true).

    - `design_blog_schema() -> list[dict]` — Return a list of table schemas for a blog: users, posts, comments. Posts has author_id FK to users. Comments has post_id FK to posts and author_id FK to users. Each table has appropriate columns, primary keys, and foreign key constraints. Constraints include: `{"type": "FOREIGN_KEY", "column": "author_id", "references": {"table": "users", "column": "id"}, "on_delete": "CASCADE"}`.

    - `normalize_table(denormalized: list[dict]) -> dict` — Given denormalized data (e.g., rows with repeated user info alongside post info), return a normalized design: `{"tables": [...], "relationships": [...]}`. The function identifies which columns repeat together and suggests separate tables with foreign keys.

    - `identify_relationship_type(table_a: str, table_b: str, description: str) -> str` — Given two tables and a description of how they relate, return "one-to-one", "one-to-many", or "many-to-many". Example: `identify_relationship_type("users", "posts", "A user can have many posts, a post belongs to one user")` → `"one-to-many"`.

    Include 6-8 pytest tests checking schema structure, required columns, foreign key references, and relationship identification.

    **3. `03_index_optimization.py` — Index Optimization**
    Functions focused on query analysis and index recommendations:

    - `suggest_indexes(table_schema: dict, queries: list[str]) -> list[dict]` — Given a table schema and a list of SQL queries, suggest which indexes to create. Analyze WHERE clauses, JOIN conditions, and ORDER BY clauses to recommend indexes. Return: `[{"table": str, "columns": list[str], "type": "INDEX" | "UNIQUE_INDEX", "reason": str}]`. Example: if query has `WHERE email = '...'` suggest index on email column.

    - `estimate_query_cost(has_index: bool, table_size: int) -> str` — Given whether an index exists and table size, return estimated query cost: "full_table_scan" (no index, any size), "index_scan" (has index, > 1000 rows), "small_table_scan" (no index, <= 1000 rows — index not needed).

    - `analyze_query_plan(query: str, indexes: list[str]) -> dict` — Given a query and existing indexes, return analysis: `{"scan_type": str, "estimated_rows": str, "uses_index": bool, "index_name": str | None, "recommendation": str}`. If the query's WHERE/JOIN column has an index, report "index_scan". Otherwise report "sequential_scan" with recommendation to add index.

    - `is_over_indexed(table_schema: dict, indexes: list[dict]) -> list[str]` — Check if a table has too many indexes. Return list of warnings (e.g., "Index on 'is_active' has low cardinality (boolean) — consider removing", "Table has 8 indexes — write performance may suffer", "Index on 'description' (TEXT) is expensive — consider partial index").

    Include 6-8 pytest tests covering index suggestion logic, cost estimation, and over-indexing detection.
