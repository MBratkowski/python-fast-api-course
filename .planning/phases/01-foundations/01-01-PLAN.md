    **Theory files (6 files):**
    Each theory file follows this structure: `# Title` → `## Why This Matters` (mobile-dev analogy) → core content with examples → `## Key Takeaways` (bullet points).

    1. `01-http-basics.md` — HTTP request/response cycle. Analogy: "You've used URLSession/Retrofit to make requests — now see what's inside those requests." Cover: client-server model, request structure (method, URL, headers, body), response structure (status line, headers, body). Show raw HTTP request/response text examples (not code yet, just protocol).

    2. `02-request-methods.md` — HTTP methods and when to use each. Cover GET, POST, PUT, PATCH, DELETE with a table showing purpose, has body, idempotent, safe. Use mobile analogy: "GET is like fetching a list view, POST is like submitting a form, PUT is like overwriting a cached object." Include examples of correct and incorrect method usage.

    3. `03-status-codes.md` — Status codes organized by family (1xx, 2xx, 3xx, 4xx, 5xx). Focus on the ones mobile devs encounter most: 200, 201, 204, 301, 304, 400, 401, 403, 404, 409, 422, 429, 500, 502, 503. Analogy: "These are like the error codes you handle in your Retrofit/URLSession completion handlers." Include a decision table: "What status code should I return when...?"

    4. `04-headers.md` — Common HTTP headers. Cover Content-Type, Accept, Authorization, Cache-Control, User-Agent, CORS headers. Analogy: "Headers are like the metadata in your URLRequest — you set Content-Type, you read Set-Cookie." Show request vs response headers. Explain content negotiation (Accept + Content-Type).

    5. `05-rest-principles.md` — REST architectural constraints: client-server, stateless, cacheable, uniform interface, layered system. Cover resource-based URLs, HATEOAS (briefly). Analogy: "REST is like how your mobile app already talks to backends — you just didn't know the formal rules." Include examples of RESTful vs non-RESTful URLs.

    6. `06-api-design.md` — API design best practices. Cover: resource naming (plural nouns, not verbs), URL hierarchy, consistent patterns, versioning (mention briefly, covered later), pagination conventions, error response format. Analogy: "This is like defining your app's navigation routes — consistent, predictable, discoverable."

    **Exercise files (3 files):**
    Module 002 is conceptual (no FastAPI yet), so exercises test understanding of HTTP concepts using pure Python functions and data structures. Each file has TODO stubs and inline pytest tests after `# ============= TESTS =============`.

    1. `01_analyze_requests.py` — Functions that parse/analyze HTTP request and response data represented as Python dicts/strings:
       - `get_request_method(request: dict) -> str` — extract method from request dict
       - `is_successful(status_code: int) -> bool` — check if status code is 2xx
       - `get_content_type(headers: dict) -> str | None` — find Content-Type header (case-insensitive)
       - `classify_status_code(code: int) -> str` — return "informational", "success", "redirect", "client_error", or "server_error"
       Include 4-5 pytest tests covering edge cases.

    2. `02_design_rest_api.py` — Functions that return correct REST API designs as data structures:
       - `design_user_endpoints() -> list[dict]` — return list of endpoint definitions (method, path, description) for a user resource CRUD
       - `design_nested_resource(parent: str, child: str) -> list[dict]` — return endpoints for nested resources (e.g., users/{id}/posts)
       - `choose_method(action_description: str) -> str` — given a description like "create a new user", return the correct HTTP method
       - `choose_status_code(scenario: str) -> int` — given a scenario like "resource created", return correct status code
       Include tests that validate correct REST conventions.

    3. `03_rest_violations.py` — Functions that identify REST violations:
       - `find_violations(endpoints: list[dict]) -> list[str]` — given a list of endpoint definitions, return descriptions of REST violations (e.g., verbs in URLs, wrong methods)
       - `fix_endpoint(method: str, path: str) -> dict` — given a non-RESTful endpoint, return the corrected version
       - `is_restful_url(url: str) -> bool` — check if URL follows REST naming conventions
       Include tests with deliberate violation examples.
