    1. `01-path-parameters.md` — Path parameters and type conversion. Cover: `{param}` syntax in route path, automatic type conversion (str, int, float, bool), `Path()` for validation constraints (ge, le, gt, lt), `Annotated[int, Path(...)]` pattern. Analogy: "Path params are like the `id` in your mobile app's navigation route — `/user/123` maps to a detail screen." Show how FastAPI auto-converts and validates.

    2. `02-query-parameters.md` — Query parameters and defaults. Cover: function params not in path become query params, default values, `Query()` for constraints and metadata, multiple query params for filtering/sorting/pagination. Analogy: "Query params are like the filter/sort options in your list screen — `/items?category=books&sort=price`." Show optional vs required, multiple values (`list[str]`).

    3. `03-request-headers.md` — Accessing and using request headers. Cover: `Header()` dependency, automatic hyphen-to-underscore conversion (x-token becomes x_token), common headers (Authorization, Content-Type, Accept, User-Agent), custom headers. Analogy: "Headers are the metadata you set on URLRequest/OkHttp — now you're reading them on the server side." Show `Header(None)` for optional, `Header(...)` for required.

    4. `04-response-models.md` — Pydantic response models. Cover: `response_model` parameter, why separate response models (hide password, add computed fields), `response_model_exclude`, response model inheritance. Analogy: "Response models are like your Codable/data class for API responses — but on the server side, you define what the client sees." Show the danger of returning raw dicts vs typed models. Introduce the Create/Response model split.

    5. `05-custom-responses.md` — Custom response classes. Cover: `JSONResponse`, `HTMLResponse`, `PlainTextResponse`, `RedirectResponse`, `StreamingResponse`, `FileResponse`. When to use each. Setting custom headers and cookies on responses. Analogy: "Like choosing between returning JSON, HTML, or a file download in your web view — different response types for different needs."

Cover: `status_code` parameter in decorators, `status.HTTP_201_CREATED` constants, `HTTPException` for error responses, custom exception handlers. Analogy: "You've been checking `response.statusCode` in your mobile app — now you're the one setting it." Show 201 for POST, 204 for DELETE, 404 for not found.

    CRITICAL: Use only Pydantic v2 patterns throughout. Never use v1 patterns:
    - Use `@field_validator` NOT `@validator`
    - Use `model_config = ConfigDict(...)` NOT `class Config:`
    - Use `model_dump()` NOT `.dict()`
    - Use `model_validate()` NOT `parse_obj()`

    **Theory files (6 files):**
    Each follows: `# Title` → `## Why This Matters` (mobile-dev analogy) → core content with code examples → `## Key Takeaways`.

    1. `01-basemodel-basics.md` — Pydantic BaseModel fundamentals. Cover: creating models by inheriting BaseModel, field declarations with types, automatic validation on instantiation, accessing fields, `model_dump()` for serialization, `model_validate()` for dict-to-model. Analogy: "Pydantic models are like Swift Codable structs or Kotlin data classes — but they validate data automatically when you create them." Show basic model creation, valid and invalid data examples, ValidationError.

    2. `02-field-validation.md` — Field types and built-in validation. Cover: `Field()` with constraints (min_length, max_length, ge, le, gt, lt, pattern), common types (str, int, float, bool, datetime, UUID, EmailStr), `Field(default=...)`, `Field(description=...)` for OpenAPI docs, `Annotated` pattern. Analogy: "Like adding @Size, @Min, @Max annotations in Kotlin, or string length validation in Swift — but declarative." Show Field constraints generating clear error messages.

    3. `03-optional-required.md` — Optional vs required fields in Pydantic v2. Cover: required fields (no default), optional nullable (`str | None = None`), required nullable (`str | None` without default — must send null), default values, `Field(default_factory=list)` for mutable defaults. Analogy: "In Kotlin, `String` is required and `String?` is nullable. Pydantic is similar but adds the 'optional to send' dimension." Show the three-way distinction: required, optional-with-default, nullable. Include common mistake: `str | None` without `= None` is still required.

    4. `04-custom-validators.md` — Custom validators with `@field_validator` and `@model_validator`. Cover: `@field_validator('field_name')` with `@classmethod`, validation modes (before/after), `@model_validator(mode='before')` and `@model_validator(mode='after')`, raising `ValueError` for validation failures, accessing other fields in model validators, `Self` type hint for after-mode model validators. Analogy: "Like custom validation rules in your mobile form validation — checking password strength, matching confirm fields." Show field validator (single field), model validator (cross-field like password confirmation). NEVER use `@validator` (v1).

    5. `05-nested-models.md` — Nested and complex model structures. Cover: models containing other models, `list[ChildModel]`, `dict[str, ChildModel]`, deeply nested structures, forward references. Analogy: "Like nested Codable structs in Swift — an Order contains a list of OrderItems, each with a Product." Show: Address nested in User, Order with list of Items, validation propagating through nested structures.

    6. `06-schema-patterns.md` — Create vs Update vs Response schema patterns. Cover: why you need separate schemas (UserCreate vs UserUpdate vs UserResponse), base model inheritance for shared fields, `UserUpdate` with all-optional fields for PATCH, `UserResponse` excluding sensitive fields (password), `model_config = ConfigDict(from_attributes=True)` for ORM integration. Analogy: "In mobile, you have separate models for API request and response — same idea, formalized." Show the complete pattern: Base → Create → Update → Response → InDB.
